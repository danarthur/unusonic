# Aion Phase A: Voice Setup + First Real Draft

_Researched: 2026-05-23 · Unusonic Research Agent_

## The question

Scope Phase A of the Aion agent architecture (section 26 of `docs/reference/follow-up-engine-design.md`). Specifically: given the Brain tab is currently paused and `public.workspaces.aion_config` doesn't exist, what's the minimum path to unblock voice setup + first real draft? Context: the goal is to have Daniel open the Brain tab, write 3 paragraphs about how he talks to clients, and immediately see an Aion-generated follow-up draft that respects that voice.

_Note: the question was written against the April 10 primer. As of May 23, several premises have changed — the findings below reflect the actual current state._

## Current state

**`aion_config` exists and is typed.** Migration `supabase/migrations/pre-baseline/20260407140000_aion_voice_foundation.sql` added `aion_config jsonb DEFAULT '{}'` to `public.workspaces`. It is fully typed at `src/types/supabase.ts:7689`. The `AionConfig` type (`aion-config-actions.ts:12–74`) has `voice` (description / example_message / guardrails), `onboarding_state`, `kill_switch`, `voice_default_derived`, and `follow_up_playbook`.

**The Brain tab is gone.** The CRM has no "Brain" tab. Aion is now a standalone page at `/aion`, rendering `ChatInterface.tsx`. The "Brain" icon at `ChatInterface.tsx:783` is a model-mode selector button ("Thinking mode"), not a tab.

**The 5-state onboarding machine is live.** `getOnboardingState()` at `aion-chat-types.ts:247` maps the config to `no_voice → no_example → no_guardrails → needs_test_draft → configured`. The chat route at `api/aion/chat/route.ts:122` calls it on every POST and drives the greeting accordingly.

**`save_voice_config` tool is wired.** `chat/tools/core.ts:118` defines the tool. When Aion captures description / example / guardrails in conversation, it calls this tool which persists to `aion_config` via `updateAionConfigForWorkspace`.

**Draft generation uses voice.** `api/aion/lib/generate-draft.ts:62–75` injects `voice.description`, `voice.example_message`, and `voice.guardrails` into the system prompt. The `draft_follow_up` chat tool at `core.ts:318` calls this with playbook rules, memory enrichment, and tone anchoring.

**Two real blockers remain:**

1. New workspaces get `voice_default_derived: true` from `aion-config-helpers.ts:43`. `getOnboardingState` short-circuits to `configured` when this flag is set (`aion-chat-types.ts:248`), so Daniel opens Aion and sees the normal configured greeting — not the "how do you talk to clients?" prompt. The reset path (sidebar overflow → "Tune Aion's voice") is buried.

2. The `needs_test_draft` step calls `getFollowUpQueue()` (`core.ts:333`). If no deals are scored yet (fresh workspace, cron hasn't run), it returns `{ error: 'No deals in the follow-up queue.' }` and the onboarding stalls with no escape path.

## Intended state

Daniel opens Aion, is prompted to describe how he communicates with clients, pastes three paragraphs (description + example + guardrails), and within that same conversation sees a follow-up draft for an active deal that sounds like him. The whole loop — voice input → draft output — happens in one chat session. The voice is then baked into every future draft generated by the follow-up engine.

## The gap

- `voice_default_derived` bypasses onboarding for every new workspace; there is no prominent first-run prompt.
- No direct "write your voice here" form — the only setup path is conversational, which is hard to aim 3 paragraphs at.
- `needs_test_draft` has no fallback when the follow-up queue is empty: the AI returns an error and the user is stuck.
- "Tune Aion's voice" lives in a sidebar overflow menu; most users will never find it.

## Options

### Option A: Voice setup form in Settings > Aion

- **What it is:** A 3-field form (description / example message / guardrails) in the existing `/settings/aion` page. Calls `saveAionVoiceConfig()` on submit. Add a "Test this voice" button that calls `POST /api/aion/draft-follow-up` directly with the current top-priority deal. Exact match for Daniel's mental model: paste 3 paragraphs, press save, see a draft.
- **Effort:** Small. No new tables, routes, or schemas. `saveAionVoiceConfig` (`aion-config-actions.ts:178`) already exists. The settings page at `settings/aion/AionSettingsView.tsx` already has `StagePanel` structure to slot into.
- **Main risk:** The "Test this voice" button depends on a deal existing. Need to handle empty-queue gracefully (show a message, don't break).
- **Unlocks:** Daniel's stated goal in full. Also unlocks a direct "update voice" flow for any owner who wants to tune without going through chat.

### Option B: First-run prompt in the chat

- **What it is:** When `voice_default_derived === true`, add a banner or a chip in the chat landing state ("Your voice hasn't been set up yet. Want to tune it now?"). Clicking calls `resetAionVoiceConfig()` inline and starts a new chat, which then hits `no_voice` and runs the 4-step conversational flow.
- **Effort:** Small. The banner renders in `ChatInterface.tsx`. `resetAionVoiceConfig` is already callable from the client (`aion-config-actions.ts:214`).
- **Main risk:** Still doesn't fix the queue-empty stall at `needs_test_draft`. The user goes through 3 conversational steps, then hits an error. Needs the queue fix from Option C to work end-to-end.
- **Unlocks:** Makes the onboarding discoverable without touching the settings page. Better fits the "everything in one chat" vision.

### Option C: Fix the queue-empty stall (required regardless)

- **What it is:** In the `draft_follow_up` tool execution (`core.ts:332–336`), when `getFollowUpQueue()` returns empty, fall back to any open deal via a simple query (`ops.events JOIN public.deals WHERE status = 'active' LIMIT 1`). Also add a "Skip test" chip to the `needs_test_draft` greeting so Daniel can mark onboarding complete without generating a draft.
- **Effort:** Small. Two changes in `core.ts` and one chip added to `buildGreeting` at `prompts.ts:329–338`.
- **Main risk:** Low. A draft generated without a queue item has no `reason` field context; the system already handles this via the `reason: 'Requested by user'` fallback at `core.ts:343`.
- **Unlocks:** Onboarding can complete on any workspace — not just ones where the cron has already run.

## Recommendation

**Do Option A + Option C together. Skip Option B for now.**

Option A is the right answer because Daniel's mental model is explicitly "write 3 paragraphs → see a draft," not "have a conversation." A form matches that intent exactly. It also surfaces voice config for owners who want to update it later — something the buried sidebar overflow never will. The implementation is small: extend `AionSettingsView.tsx` with three `<textarea>` fields, call the existing `saveAionVoiceConfig` action, and add a "Test draft" button that fires `POST /api/aion/draft-follow-up` and renders the result inline.

Option C must ship alongside it. Without fixing the queue-empty case, the "Test draft" button in the form will break on fresh workspaces. The fix is two small changes in one file.

Option B (chat banner) can come later as a quality-of-life improvement. The conversational onboarding still works fine for users who prefer it — the form is additive, not a replacement.

4. In `src/app/api/aion/chat/route/prompts.ts:329–338` (the `needs_test_draft` greeting), add a third chip: `{ label: 'Skip for now', value: 'I am done setting up.' }`. The AI will call `save_voice_config` with `onboarding_complete: true` when the user picks it.
5. Smoke-test: reset voice config via sidebar, walk the 4-step chat flow, confirm `aion_config.voice` is populated in Supabase, then click "Test draft" from Settings and verify voice appears in the generated message.
6. If the Settings form route feels heavy, the minimum viable version is just the 3 textareas + save — the "Test draft" button can ship in the following sprint.
